# ms-timer — 18시 정각까지 1ms 카운트다운 플립 시계

**Date:** 2026-07-15
**Status:** Approved (rev.2 — 플립 시계 그래픽 반영)

## 목적

근무 종료(18:00:00.000)까지 남은 시간을 1ms 단위로 실시간 표시한다. 시간의 흐름을 시각적으로 체감시키는 것이 유일한 목적이며, 다른 기능은 없다.

## 설계 전제 1: 60Hz 현실

모니터 refresh rate는 60Hz(16.7ms) 또는 144Hz(6.9ms)다. **ms 단위 3자리는 물리적으로 개별 프레임을 볼 수 없다.**

이는 버그가 아니라 제품의 본질이다. 설계는 이 한계를 숨기지 않고 정면으로 활용한다.

## 설계 전제 2: 플립 애니메이션의 시간 예산

플립 애니메이션 1회는 최소 250~300ms가 필요하다. 그보다 짧으면 "종이가 넘어가는 것"으로 지각되지 않고 깜빡임이 된다. 자릿수별 변경 주기와 대조하면:

| 자리 | 변경 주기 | 플립 |
|---|---|---|
| 시 | 1시간 | 가능 |
| 분 | 60초 | 가능 |
| 초 | 1000ms | 가능 (300ms 플립이 여유롭게 들어감) |
| ms 백의자리 | 100ms | 불가 (80ms 플립은 지각되지 않음) |
| ms 십의자리 | 10ms | 불가 (60Hz에서 프레임 0.6개) |
| ms 일의자리 | 1ms | 불가 |

**따라서 플립은 시:분:초까지다. ms 3자리는 구조적으로 플립할 수 없다.**

이 제약이 오히려 제품을 완성시킨다. 플립 카드가 착착 넘어가는 옆에서 ms가 읽을 수 없는 속도로 뭉개지면, "시간이 흐른다"와 "시간이 미친듯이 흐른다"가 한 화면에서 대비된다. 이것이 목적에 정확히 부합한다.

## 핵심 로직 1: 절대 시각 차분

```js
remaining = todayAt18(now).getTime() - now.getTime()
```

누적 방식(`remaining -= 16`)이 아닌 **절대 시각 차분**이다. 이 선택 하나로 다음이 전부 공짜로 처리된다:

- 드리프트 원천 차단 (누적 오차 없음)
- 절전 → 복귀 (누적 방식이면 슬립 3시간이 통째로 어긋남)
- NTP 시계 보정 / 시스템 시계 변경
- 타임존 변경

### 인터페이스

```js
computeRemaining(now: Date, targetHour = 18)
  → { expired: boolean, h: number, m: number, s: number, ms: number }
```

- `expired`가 `true`면 나머지 필드는 전부 `0`
- 입력 `now`를 **변경하지 않는다** (순수 함수)

## 핵심 로직 2: 릴 위상

릴은 이산적 값이 아니라 **연속 위상**으로 돈다. 그래야 60Hz 샘플링에서 자연스러운 잔상이 생긴다.

```js
reelPhase(ms: number, place: 100 | 10 | 1) → number   // 0 이상 10 미만
```

`(ms % (place * 10)) / place` — 한 줄이다. 자리별 회전 속도:

| place | 위상 주기 | 초당 회전 | 지각 |
|---|---|---|---|
| 100 | 1000ms | 1바퀴 | 읽힐 듯 말 듯 |
| 10 | 100ms | 10바퀴 | 번짐 |
| 1 | 10ms | 100바퀴 | 순수 블러 |

렌더링: `translateY(-phase * 자릿높이)`.

카운트다운이므로 `ms`가 999→0으로 감소하고 위상도 함께 감소한다. 릴 스트립은 위에서 아래로 `0..9`이므로, 위상 감소는 릴이 아래로 흐르며 숫자가 9→0으로 줄어드는 것으로 보인다. 방향이 맞다.

**블러 보강:** 십/일의자리는 60Hz 샘플링으로는 위치가 프레임마다 튄다(일의자리는 초당 1000회 변하는데 60회만 샘플링). 이는 매끄러운 잔상이 아니라 지터로 보인다. 자리별로 `filter: blur()`를 고정 적용해 이를 모션 블러로 지각되게 한다 — 백의자리 0, 십의자리 1px, 일의자리 3px. 위치는 실제 값에서 계산되므로 블러는 속임수가 아니라 물리적 사실의 시각화다.

## 플립 카드

자릿수별로 독립된 카드 6개(`HH:MM:SS`). 각 카드는 **자기 값이 바뀔 때만** 플립한다 — 초의 십의자리는 10초마다, 시의 자리는 1시간마다. 실제 split-flap 기계와 동일한 동작이다.

표준 CSS 3D 기법: 카드를 가로로 반 접고, 이전 값의 상단 절반이 `rotateX(0 → -90deg)`로 아래로 접히는 동안(0~50%), 새 값의 하단 절반이 `rotateX(90deg → 0)`로 펼쳐진다(50~100%). `transform`만 애니메이션하므로 GPU 합성되어 비용이 없다.

- 플립 지속시간: 300ms
- 최대 동시 플립: 6장 (17:59:59 → 18:00:00 같은 경계) — 60fps에서 무부하

## 테마

CSS 변수로만 분리한다. `themes.css`가 팔레트를, `style.css`가 구조를 담당하며 서로 침범하지 않는다.

- **다크 모던** (기본): 검은 배경 / 짙은 회색 카드 / 흰 숫자
- **레트로 민트**: 민트 프레임 / 크림 카드 / 딥그린 숫자

**레트로 테마의 범위:** 색상 팔레트만 재현한다. 참조 이미지의 금속 스탠드·다리·질감은 만들지 않는다 — CSS로 구현하면 앱 본체보다 커지며 목적(시간 체감)에 기여하지 않는다.

토글은 `T` 키다. 버튼을 넣으면 "다른 기능 없이" 원칙이 무너지고, 클릭 토글은 오조작 위험이 있다.

**영속화:** `localStorage`에 저장하여 다음 실행 시 복원한다.

**실측 완료 (Task 5):** Electron 43은 `file://` origin에서 localStorage를 **허용한다.** 레트로 상태로 종료 후 재실행하면 레트로로 복원된다.

코드는 `try/catch`로 감싼 채로 둔다. 차단되는 환경에서는 조용히 세션 전용 토글로 떨어진다 — 기능은 유지된다. 이는 죽은 코드가 아니라 문서화된 설계 결정이며, IPC/preload 도입보다 낫다: 테마 기억 하나 때문에 프로세스 간 통신 계층을 추가할 가치가 없다.

## 렌더링

`requestAnimationFrame` 루프 하나가 매 프레임 `computeRemaining()` → 카드 6개 + 릴 3개 갱신을 수행한다.

`setInterval(1ms)`는 쓰지 않는다 — 브라우저가 clamp하며, rAF보다 자주 그려도 모니터가 표시하지 못한다.

### 필수 요건

- **`backgroundThrottling: false`** — 창이 가려지면 Chromium이 rAF를 멈춘다
- **플립 트리거는 값 변경 시에만** — 매 프레임 애니메이션을 재시작하면 카드가 영원히 안 넘어간다
- **릴은 매 프레임 `transform` 갱신** — 연속 회전이므로 값 비교가 무의미하다

`tabular-nums`는 더 이상 필요 없다. 자릿수마다 고정폭 카드/릴에 들어가므로 레이아웃이 구조적으로 고정된다.

## 보안

Electron 기본값 유지: `nodeIntegration: false`, `contextIsolation: true`.
renderer는 `Date.now()`와 `localStorage`만 사용하므로 preload / IPC가 불필요하다.

**CSP meta 태그는 넣지 않는다.** 원격 콘텐츠를 0건 로드하고, 네트워크 요청이 없으며, node 통합이 꺼져 있다. CSP의 한계 이득 대비 `file://` origin에서 `'self'` 지시어가 ESM import를 막는 마찰이 크다. 이 결정은 "원격 콘텐츠를 로드하지 않는다"는 전제에 의존하므로, 네트워크 기능이 생기면 CSP를 반드시 추가해야 한다.

## 테스트 전략

`lib/`의 순수 함수만 테스트한다. `renderer/`에는 로직이 없다.

### countdown.js

| 케이스 | 입력 | 기대 |
|---|---|---|
| 정상 카운트다운 | 09:00:00.000 | `{expired:false, h:9, m:0, s:0, ms:0}` |
| 18시 1ms 전 | 17:59:59.999 | `{expired:false, h:0, m:0, s:0, ms:1}` |
| 18시 정각 | 18:00:00.000 | `{expired:true, ...0}` |
| 18시 1ms 후 | 18:00:00.001 | `{expired:true, ...0}` |
| 18시 이후 실행 | 19:30:00.000 | `{expired:true, ...0}` |
| 자정 직후 | 00:00:00.000 | `{expired:false, h:18, m:0, s:0, ms:0}` |
| 자릿수 혼합 | 08:07:06.005 | `{h:9, m:52, s:53, ms:995}` |
| targetHour 파라미터화 | 17:00, target 19 | `{h:2, m:0, s:0, ms:0}` |
| 순수성 | 임의 Date | 입력이 변경되지 않음 |
