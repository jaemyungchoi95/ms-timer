# 퇴근시간 사용자 설정 — 설계

날짜: 2026-07-16
대상: `ms-timer`
선행 스펙: `2026-07-15-ms-timer-design.md` (플립 클럭 rev.2)

## 배경

기존 스펙은 퇴근 시각을 18:00으로 고정했다. 그러나 회사 유연근무제는 출근이 07:30–10:00이고 9시간 근무 후 퇴근이라, **실제 퇴근 시각이 16:30–19:00 사이에 사람마다·날마다 다르다.** 18:00이 하드코딩된 앱은 사용자의 절반에게 틀린 숫자를 보여준다.

이 앱의 목적은 "오늘은 언제까지 일하고 퇴근할 건데, 시간이 안 가네"라고 느낄 때 켜는 것이다. 그 문장의 "언제까지"가 사람마다 다르다면, 그건 설정값이어야 한다.

### 원 스펙 문구 정정

원 스펙의 "다른 기능은 없이 그냥 18시 정각 − 현재 시간"은 두 번 완화되었다.

1. rev.2에서 테마 토글 (`T` 키)
2. 본 스펙에서 퇴근시간 설정

의도는 여전히 유효하다 — **보조 정보를 늘리지 않는다.** 경과율, 진행바, 시계, 날짜는 넣지 않는다. 추가되는 것은 카운트다운의 **입력**이지 카운트다운 옆의 **장식**이 아니다. 이 구분이 앞으로의 기능 판단 기준이다.

## 설계 전제

선행 스펙의 두 전제(60Hz 물리 한계 = 제품의 목적, 플립 애니메이션 250–300ms 하한)는 그대로 유효하며 본 스펙이 바꾸지 않는다.

본 스펙이 더하는 전제:

**퇴근시간은 매일 바뀌는 값이다.** 유연근무제에서 오늘의 퇴근 시각은 오늘의 출근 시각이 정한다. 따라서 영속화의 의미는 "한 번 설정하면 끝"이 아니라 **"마지막 값을 다음 실행의 기본값으로 기억"** 이다.

이 값은 하루 한 번 만져질 것을 전제로 설계한다 — 설정 화면이나 메뉴에 묻으면 안 되고, 메인 화면에서 클릭 한 번에 편집 상태가 되어야 한다. 하루 한 번 쓰는 컨트롤과 한 달에 한 번 쓰는 컨트롤은 크기도 위치도 다르다. 대부분의 날에는 어제와 같은 값이라 그대로 두고 넘어가므로, 기본값이 채워져 있는 것이 조작 횟수를 결정한다.

## 결정 사항

| 항목 | 선택 | 근거 |
|---|---|---|
| 입력 범위 | 00:00–23:59, 분 단위 | 사용자 결정 |
| 과거 시각을 target으로 확정 | 즉시 "퇴근" 표시 (내일로 롤오버 안 함) | 아래 "자정 롤오버를 하지 않는 이유" |
| 검증 방식 | 자유 입력 + 실시간 무효 표시 (✓ 비활성 + 빨간 테두리) | 자리별 키 차단은 09:00을 치려고 `9`를 먼저 누른 사용자에게 무반응으로 보인다 |
| ✓ 버튼 | 확정 — 저장 + 타이머 재설정 | |
| ↻ 버튼 | 편집 취소 — 원래 값 복귀, 편집창 닫힘 | 유연근무제에서 18:00은 특별한 값이 아니라 "기본값으로 리셋"은 쓸 일이 없다. 오타·변심에 의한 취소가 훨씬 잦다 |
| 키보드 | Enter = ✓, Esc = ↻ | |
| 영속화 | `localStorage`, 키 `ms-timer:target`, 값 `"HH:MM"` | `theme.js`가 확립한 패턴을 따른다. 값을 사람이 읽을 수 있는 형식으로 두어 DevTools에서 바로 확인·수정 가능하다 |
| 하단 라벨 | 제거, 상단 헤더로 이동 | 사용자 UI 시안 |

### 자정 롤오버를 하지 않는 이유

"target이 과거면 내일로 넘긴다"는 야간 교대근무를 지원하지만 **퇴근 표시 기능을 죽인다.** 18:00에 퇴근한 사람이 18:01에 앱을 켜면 "퇴근" 대신 23시간 59분 카운트다운을 보게 된다. 퇴근 표시는 rev.2에서 사용자가 직접 고른 동작이다.

야간 교대근무자는 이 앱의 대상이 아니다. 새벽까지 야근할 사람은 이 앱을 켜지 않는다. **정상 케이스를 망가뜨려 대상 아닌 사용자를 지원하는 것은 손해다.**

부수 효과로, 이 결정 덕에 "앱을 켠 사람의 target은 정의상 미래"라는 전제가 성립하여 롤오버를 고민할 상황 자체가 발생하지 않는다.

## 아키텍처

선행 스펙의 경계를 유지한다: **`src/lib/`는 DOM을 만지지 않고 모든 로직과 모든 테스트를 갖는다. `src/renderer/`는 로직이 없고 설계상 테스트도 없다.** 이 경계는 rev.2에서 그래픽을 전면 교체할 때 `countdown.js`가 한 글자도 바뀌지 않는 것으로 이미 값을 증명했다.

```
src/lib/                순수 · 테스트 있음
  countdown.js          computeRemaining(now, target = {h: 18, m: 0})   ← 시그니처 변경
  target-time.js        NEW — parseTarget / formatTarget
  reel-phase.js         변경 없음

src/renderer/           표현 · 테스트 없음
  clock.js              target 보유, tick에 주입
  target-editor.js      NEW — 표시/편집 UI + localStorage
  index.html            헤더 추가, 하단 라벨 제거
  style.css             헤더/편집기
  themes.css            --reel-fg 수정, --label-fg → --title-fg 개명, 편집기 색 추가
  theme.js              T 키 가드 추가
  flip-digit.js         변경 없음
  reel.js               변경 없음
```

### 시그니처 변경

```js
// before
export function computeRemaining(now, targetHour = 18)
// after
export function computeRemaining(now, target = { h: 18, m: 0 })
```

위치 인자(`computeRemaining(now, 18, 0)`)를 쓰면 기존 테스트가 안 깨지지만, target은 저장·검증·포맷·표시에서 **항상 한 덩어리로 다닌다.** 인자 둘로 쪼개면 호출부마다 다시 묶어야 한다. 테스트 한 줄을 아끼려고 개념을 쪼개지 않는다 — 개념 자체가 "목표 시"에서 "목표 시각"으로 바뀌었으므로 테스트가 바뀌는 것이 정직하다.

#### 지역 변수 이름 충돌 — 반드시 함께 고칠 것

현재 `countdown.js`는 목표 시각 `Date`를 담는 지역 변수를 `target`이라 부른다. 파라미터도 `target`으로 두면 **재선언 SyntaxError**가 난다.

```js
export function computeRemaining(now, target = { h: 18, m: 0 }) {
  const target = new Date(now);   // ✗ SyntaxError: 'target' has already been declared
```

지역 변수를 `deadline`으로 개명한다.

```js
export function computeRemaining(now, target = { h: 18, m: 0 }) {
  const deadline = new Date(now);          // now의 복사본 — 원본 불변 (순수성 테스트가 이를 고정)
  deadline.setHours(target.h, target.m, 0, 0);
  const diff = deadline.getTime() - now.getTime();
  // 이하 동일
}
```

`new Date(now)` 복사 생성자는 그대로 유지한다. `now.setHours()`를 직접 부르면 입력 `Date`를 변경하여 순수 함수 테스트(`countdown.test.js:47`)가 깨진다.

절대 시각 차분 방식은 그대로이므로 drift 면역·절전/복귀·NTP 보정·타임존 변경은 계속 공짜로 따라온다.

### 데이터 흐름

1. 부팅: `target-editor`가 localStorage를 읽어 `parseTarget`에 넘긴다 → null이 아니면 사용, null이면 `{h: 18, m: 0}`
2. `clock.js`가 현재 target을 보유하고, 매 프레임 `computeRemaining(new Date(), target)`
3. 사용자가 목표 시각 표시를 클릭 → 편집 모드 → ✓ → `onChange({h, m})` → `clock.js`의 target 교체 + localStorage 기록
4. 다음 프레임부터 새 target 기준

**플립 카드에 별도 처리가 필요 없다.** 기존 상태 머신은 값의 변화를 보고 넘긴다. target이 바뀌어 남은 시간이 점프하면 해당 자릿수 카드들이 자연스럽게 넘어간다.

### 순수 함수 계약

```js
// src/lib/target-time.js

/**
 * "HH:MM" → {h, m}. 형식이 다르거나 00:00–23:59 범위를 벗어나면 null.
 * 형식은 /^(\d{2}):(\d{2})$/ 로 엄격 매칭한다.
 */
export function parseTarget(str)

/** {h, m} → "HH:MM" (2자리 zero-pad) */
export function formatTarget(target)
```

두 함수는 왕복 대칭이다: 유효한 `s`에 대해 `formatTarget(parseTarget(s)) === s`.

**이 API가 검증 경로를 하나로 합친다.** 저장값 검증과 사용자 입력 검증이 같은 함수를 탄다 — 편집기는 네 칸을 `` `${d0}${d1}:${d2}${d3}` ``로 합쳐 `parseTarget`에 넘기고, 부팅 시 localStorage 값도 그대로 `parseTarget`에 넘긴다. 검증 규칙이 한 곳에만 존재하므로 두 경로가 어긋날 수 없고, 같은 테스트가 양쪽을 모두 덮는다.

**엄격 매칭이 빈 칸 함정을 자동으로 막는다.** 한 칸이 비면(`[ ][8]:[3][0]`) 합쳐진 문자열이 `"8:30"`이 되어 정규식이 실패하고 null이 된다. 느슨하게 파싱했다면 `parseInt("8")`이 `8`을 반환해 `"08:30"`으로 **조용히 유효 판정**되었을 것이다 — 시각적으로는 미완성인데 ✓가 활성화되는 버그다.

범위 검사(`24:00`, `23:60`)도 `parseTarget`이 함께 수행하여 null을 반환한다. 별도의 `isValidTarget`을 두지 않는 이유는, 두면 "형식은 맞지만 범위가 틀린" 중간 상태가 생겨 호출부마다 두 번 검사해야 하기 때문이다.

### 입력 방식

네 개의 한 자리 입력 칸이다. 플립 카드가 자릿수마다 한 장인 것과 같은 언어를 쓴다.

- 각 칸은 숫자 한 글자만 받는다. 숫자가 아닌 키는 무시한다
- 입력 시 다음 칸으로 자동 이동, 마지막 칸에서는 머무른다
- Backspace: 칸이 비어 있으면 이전 칸으로 이동
- ←/→ 로 칸 간 이동
- 포커스 시 해당 칸의 값을 전체 선택하여 타이핑이 곧 덮어쓰기가 되도록 한다

**자리별 키 차단은 하지 않는다.** 시 첫 칸에 `0–2`만 받는 식으로 막으면 09:00을 치려고 `9`를 먼저 누른 사용자에게 무반응으로 보인다. 어떤 숫자든 받고, 결과가 무효면 표시한다.

### 편집기 상태 기계

상태는 둘뿐이다.

- **표시 모드** — `18:00` 텍스트, 클릭 가능
- **편집 모드** — `[1][8]:[0][0] ✓ ↻`, 첫 칸 자동 포커스 + 전체 선택

전이: 클릭 → 편집. ✓/Enter → 유효하면 확정 후 표시. ↻/Esc → 파기 후 표시.

무효는 별도 상태가 아니라 편집 모드의 렌더링 속성이다. 매 입력마다 `parseTarget`이 null인지로 판정해 ✓의 `disabled`와 테두리 색을 갱신한다.

**무효 표시는 칸 단위가 아니라 그룹 단위다.** `parseTarget`은 null 하나만 반환하므로 어느 칸이 틀렸는지 알려주지 않는다. 이는 의도된 단순화다 — `25:70`처럼 두 칸이 동시에 틀린 경우가 흔하고, 네 칸이 하나의 시각을 이루므로 "이 시각이 무효"가 "이 칸이 무효"보다 정확한 진술이다.

## 레이아웃

```
┌────────────────────────────────────┐
│         퇴근까지          18:00     │  ← 헤더: 제목 중앙, 목표 시각 우측
│  ┌──────────────────────────────┐  │
│  │  0 8 : 2 8 : 3 5 . ▓▓▓       │  │  ← 플립 프레임 (기존)
│  └──────────────────────────────┘  │
└────────────────────────────────────┘
```

헤더 행은 플립 프레임과 같은 폭을 갖고, 제목은 중앙, 목표 시각은 우측에 정렬한다. 하단 라벨은 제거한다.

**expired 시 헤더 제목이 "퇴근까지" → "퇴근"으로 바뀐다.** 현재 이 역할은 하단 라벨(`clock.js:34`)이 겸하고 있으므로, 라벨 제거 시 역할이 헤더로 이동한다. `.clock.expired` 하위 색상 규칙도 함께 옮긴다.

### 테마 변수

**"제목은 흰색"은 dark 테마에서만 성립한다.** 헤더는 프레임이 아니라 `--bg` 위에 놓이는데, 레트로의 `--bg`는 민트색(`#8fc4b2`)이라 흰 글자는 대비 약 1.9:1로 **릴 버그와 같은 종류의 비가시 상태**가 된다. 테마별로 값을 나눈다.

`--label-fg`는 하단 라벨과 함께 사라지므로 `--title-fg`로 개명한다 (제목과 목표 시각 표시가 공용). 크기 차이가 위계를 지고 색은 공유한다.

| 변수 | dark | retro | 용도 |
|---|---|---|---|
| `--title-fg` | `#f7f8f8` | `#2c4a3e` | 헤더 제목 + 목표 시각 표시 |
| `--reel-fg` | `#6b7280` (유지) | `#2c4a3e` (**수정**) | ms 릴 — 버그 수정 |
| `--input-bg` | `#1c1f24` | `#f4f0e4` | 입력 칸 배경 (각 테마의 카드색) |
| `--input-fg` | `#f7f8f8` | `#2c4a3e` | 입력 칸 글자 (각 테마의 카드 글자색) |
| `--input-border` | `#3a3f46` | `#4f8f7c` | 입력 칸 테두리 (정상) |
| `--input-border-invalid` | `#ef4444` | `#c0392b` | 입력 칸 테두리 (무효) |
| `--btn-fg` | `#6b7280` | `#2c4a3e` | ✓ / ↻ 버튼 |

입력 칸은 각 테마의 **카드 색을 재사용한다** — 편집 UI가 플립 카드와 같은 재질로 읽히고, 새 색을 발명하지 않는다.

무효 테두리만 팔레트 밖의 빨강을 쓴다. 오류는 테마 조화보다 눈에 띄는 것이 우선이고, 레트로는 채도를 낮춘 `#c0392b`로 민트 배경과 충돌을 줄인다.

## 엣지 케이스

| 케이스 | 처리 |
|---|---|
| 저장값 없음 / 손상 / 범위 밖 | `parseTarget`이 null → `{h: 18, m: 0}` 폴백. 사용자 입력과 **같은 검증 함수**를 타므로 두 경로가 어긋날 수 없다. `theme.js`의 화이트리스트 검증과 같은 방어 사상 |
| localStorage 자체가 차단된 환경 | try/catch로 세션 전용 동작. `theme.js` 주석 참조 — Electron 43 `file://`에서 영속화됨이 2026-07-15 실측 확인됨 |
| 빈 칸 / 범위 밖 입력 | ✓ 비활성 + 빨간 테두리. Esc·↻로는 언제든 탈출 가능 |
| 과거 시각 확정 | 즉시 "퇴근" |
| **expired 중 목표 시각 변경** | 미래로 바꾸면 카운트다운 재개. 야근 연장 시 유용하며 설계상 공짜로 따라온다 |
| **편집 중 `T` 키** | 현재 `theme.js`가 `window`에 keydown을 걸어 **입력창에서 T를 치면 테마가 바뀐다.** 포커스가 입력 요소에 있으면 무시하도록 가드 |
| **`body { user-select: none; cursor: default }`** | 현재 전역이라 **입력칸에서 드래그 선택이 안 되고 커서가 화살표로 뜬다.** 입력칸만 예외 처리 |
| 편집 중 타이머 | rAF는 계속 돈다. 편집이 카운트다운을 멈추지 않는다 |

## 버그 수정 — 레트로 테마 ms 릴 비가시

### 증상

레트로 테마에서 ms 3자리가 보이지 않는다.

### 근본 원인

릴은 정상 동작한다. 위상 계산도 transform도 멀쩡하다. **그려지는데 안 보이는 것이다.**

```
retro:  --reel-fg: #6d8a7d  on  --frame-bg: #4f8f7c   →  대비 1.005 : 1
dark:   --reel-fg: #6b7280  on  --frame-bg: #15181d   →  대비 3.68  : 1
```

레트로의 릴 색과 프레임 배경색은 상대 휘도가 사실상 동일하다 (G 138 vs 143, B 125 vs 124). WCAG 최소 대비가 큰 글자 기준 3:1인데 1.005:1이므로 육안 식별이 불가능하다. 사용자 스크린샷에서 숫자 윤곽이 희미하게 보이는 것이 "렌더링은 되고 있다"는 증거다.

`.reel`은 자체 배경이 없고 `.frame` 안에 있으므로 릴 숫자는 `--frame-bg` 위에 직접 그려진다.

### 토글은 원인이 아니다

릴 JS는 색을 만지지 않고 테마는 CSS 변수다. 처음부터 레트로로 켰어도 동일하게 안 보였다. `T` 토글 로직이 아니라 **레트로 테마의 색 값** 문제이므로 수정 지점은 `themes.css` 한 줄이다.

### 수정

`--reel-fg`를 `#6d8a7d` → `#2c4a3e` (테마의 기존 잉크색, 대비 2.58:1).

방향은 **어둡게**다. 레트로 테마에서 초록 프레임 위의 글자는 이미 전부 어두운 잉크(`--sep-fg`, `--label-fg` 모두 `#2c4a3e`)이며, 크림 카드 + 어두운 잉크는 실물 플립시계의 관례다. 릴만 밝게 하면 초록 위 밝은 글자가 릴 하나뿐이라 따로 논다.

2.58:1은 dark 테마의 3.68:1보다 약간 흐릿하지만 **의도된 방향이다.** 릴은 카드보다 흐릿해야 한다 — 판독 불가가 제품의 핵심이다. 다만 "흐릿함"과 "비가시"는 다르며, 현재 값은 후자다.

구현 후 실제 화면 스크린샷으로 확인하고 필요하면 미세조정한다.

**`target-time.test.js`** — 신규

`parseTarget` 유효:
- `"00:00"` → `{h: 0, m: 0}` (하한 경계)
- `"23:59"` → `{h: 23, m: 59}` (상한 경계)
- `"16:30"` → `{h: 16, m: 30}` (실사용 값)

`parseTarget` 범위 밖 → null:
- `"24:00"` (시 상한 초과), `"23:60"` (분 상한 초과)

`parseTarget` 형식 불일치 → null:
- `"8:30"` — **빈 칸 함정.** 느슨한 파싱이었다면 `8`로 통과했을 값
- `"16:3"`, `":"`, `""` — 부분 입력
- `"1630"` — 콜론 없음
- `"ab:cd"` — 숫자 아님
- `"-1:00"` — 부호. 정규식이 `-`를 거부함을 고정
- `"16:30 "` / `" 16:30"` — 공백. `^$` 앵커가 있음을 고정
- `null`, `undefined`, 숫자 타입 — 문자열이 아닌 입력

`formatTarget`:
- `{h: 9, m: 5}` → `"09:05"` (zero-pad)
- `{h: 0, m: 0}` → `"00:00"`
- `{h: 23, m: 59}` → `"23:59"`

왕복 대칭:
- 유효한 문자열에 대해 `formatTarget(parseTarget(s)) === s`
